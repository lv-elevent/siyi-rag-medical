# Claude Code 编写规范（Medical RAG Agent Project）

本文档用于规范 Claude Code 在本项目中的代码生成与修改行为，目标是：

- 保持项目架构稳定
- 避免不必要的大重构
- 保证分阶段开发可控
- 让代码结果更适合后续人工收口、测试和简历包装

---

## 1. 项目基本原则

### 1.1 最小侵入式改造
Claude Code 在修改项目时，必须优先遵守：

- 尽量少改已有模块
- 优先新增文件，而不是重写旧文件
- 若现有接口不统一，应在新增模块中做适配
- 不要为了实现一个新功能，大面积重构整个项目

### 1.2 分阶段开发
本项目按阶段推进，每次只实现当前阶段要求的内容。

Claude Code 必须：
- 只完成当前阶段任务
- 不提前实现后续阶段能力
- 不擅自增加复杂设计

例如：
- 阶段 A 只做 Agent 基础骨架
- 不应提前加入 memory、followup、cross-encoder、复杂前端改造

### 1.3 可运行优先
代码生成优先级：
1. 可运行
2. 与现有架构兼容
3. 易于扩展
4. 再考虑抽象优雅

不要为了“形式上的完美抽象”牺牲可运行性和兼容性。

---

## 2. 项目架构约束

当前项目采用分层结构：

- `backend/api/`：接口层
- `backend/models/`：请求/响应模型
- `backend/services/`：业务逻辑层
- `backend/repositories/`：数据访问层
- `backend/core/`：配置与基础客户端

Claude Code 必须遵循以下规则：

### 2.1 API 层职责
- 只负责请求接收、参数校验、调用 service、返回结果
- 不要在 `api/` 中写复杂业务逻辑

### 2.2 Services 层职责
- 负责业务编排和核心逻辑
- 新能力优先在 `services/` 中实现
- 如果是统一流程编排，优先封装为 service

### 2.3 Models 层职责
- 只定义数据结构
- 不要在模型文件中写业务逻辑

### 2.4 Repositories 层职责
- 只做向量库/数据库访问封装
- 不要在 repository 中写上层业务逻辑

### 2.5 Core 层职责
- 负责配置、客户端、日志
- 不要把业务逻辑塞到 core 里

---

## 3. 代码修改原则

### 3.1 优先新增，不轻易重写
如果要接入新能力：
- 优先新增文件或新增类
- 谨慎修改已有主流程
- 若需改已有文件，尽量局部修改

### 3.2 不擅自删除旧逻辑
Claude Code 不应删除旧代码路径，除非明确要求。

例如：
- 接入 Agent 时，应保留原有非 Agent 调用路径
- 新旧逻辑可以并存，通过开关控制

### 3.3 保持兼容
新增字段、参数、返回结构时：
- 尽量保证向后兼容
- 避免破坏现有接口调用方式

### 3.4 不要虚构复杂依赖
如果项目中没有某个依赖：
- 不要擅自引入大而重的新框架
- 除非明确要求，否则优先使用现有依赖和轻量规则实现

---

## 4. 阶段开发规范

### 4.1 严格只做当前阶段内容
Claude Code 在每次任务中，只做当前阶段要求的事情。

例如：
- 阶段 A：Agent 基础骨架
- 阶段 B：memory + 主动追问
- 阶段 C：Query 理解增强
- 阶段 D：Cross-encoder 精排
- 阶段 E：结构化回答 + 安全增强
- 阶段 F：联调、文档、展示

不得提前实现：
- Redis 持久化
- 知识图谱
- 多模态
- 复杂前端框架
- 训练模型
- 与当前阶段无关的功能

### 4.2 当前阶段不确定时，先适配而非重构
如果当前阶段要调用旧 service，但旧接口风格不一致：
- 优先在新增模块中封装适配
- 不要反向重写所有旧 service

---

## 5. 输出要求规范

Claude Code 每次生成结果时，必须输出：

1. 修改后的目录结构
2. 新增/修改文件的完整代码
3. 调用链路说明
4. 兼容性说明
5. 风险点说明（如果存在接口猜测或适配假设）

如果没有完整代码输出，只给片段，不符合要求。

---

## 6. 代码风格规范

### 6.1 保持与现有项目风格一致
- 命名风格保持统一
- 导入风格保持统一
- 使用项目现有框架和模式

### 6.2 注释要克制
- 对关键编排逻辑可加简要注释
- 不要写冗长空洞注释
- 不要生成大量模板化废话注释

### 6.3 不做伪抽象
避免无意义的：
- 工厂模式套工厂
- 过度接口抽象
- 为了“优雅”而增加复杂层级

当前项目优先要的是：
- 清晰
- 可运行
- 易维护
- 易讲解

---

## 7. 错误处理规范

Claude Code 生成的代码应具备基本健壮性：

- 检索为空时不要直接崩溃
- source 提取失败时要有默认值
- 中间 service 调用失败时要尽量降级
- 不要让单点异常导致整个链路 500

---

## 8. 不允许的行为

Claude Code 不应：

- 擅自重构整个项目目录
- 一次性修改大量无关文件
- 未经要求引入复杂第三方框架
- 删除旧接口或旧逻辑
- 提前实现后续阶段功能
- 只输出思路，不输出完整代码
- 用伪代码替代可运行代码
- 对未知接口做武断假设而不标注说明

---

## 9. 推荐行为

Claude Code 应优先：

- 做适配层
- 保持新旧逻辑兼容
- 留好后续扩展点
- 输出清晰的代码和说明
- 在不确定处明确标注“基于现有接口假设”
- 尽量减少后续人工返工成本

---

## 10. 本项目开发目标提醒

本项目目标不是做学术研究原型，而是做一个：

**适合简历展示、可运行、可讲解、具备亮点的医疗 RAG Agent 工程项目**

因此 Claude Code 生成代码时，应始终围绕以下目标：

- 工程化
- 可展示
- 亮点清晰
- 适合 1–2 周内快速完成
- 适合后续面试介绍与项目包装

---

## 11. 每次任务前的默认执行原则

Claude Code 在收到每次阶段任务时，默认遵守：

1. 只做当前阶段要求
2. 优先最小侵入
3. 保留旧逻辑兼容
4. 输出完整代码
5. 对不确定处标注假设
6. 不擅自扩大改动范围
